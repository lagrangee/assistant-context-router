# Escalation Protocol

> 本协议定义：什么时候必须升级给 project owner，什么时候 coordinator agent 可以自己决策。
>
> 核心原则：**human in the loop, not as the bus**

---

## 升级矩阵（Escalation Matrix）

| 场景 | 升级级别 | 说明 |
|------|----------|------|
| **方向性决策** | 🔴 必须升级 | 项目目标、范围、成功标准变更 |
| **边界拍板** | 🔴 必须升级 | 技术选型、架构决策、外部依赖引入 |
| **资源/权限** | 🔴 必须升级 | 需要 project owner 授权执行（如脚本、配置变更、外发） |
| **验收判断** | 🔴 必须升级 | 阶段完成是否达标，能否进入下一阶段 |
| **异常处理** | 🟡 条件升级 | 未定义场景，需判断是否有先例可循 |
| **文档收口** | 🟢 coordinator agent 自主 | 按已确认的结构和格式，整理、更新、迁移 |
| **agent 调度** | 🟡 条件升级 | 外部 agent 进入项目工作，需先确认 scope 和边界 |
| **工作态维护** | 🟢 coordinator agent 自主 | RESUME、COLLAB、STATUS 的日常更新 |

---

## 详细规则

### 🔴 必须升级给 project owner 的情况

#### 1. 方向性决策
- 项目目标变更
- 成功标准重新定义
- 阶段划分调整
- 范围扩张或收缩

#### 2. 技术/架构边界拍板
- 引入新的技术栈或依赖
- 改变现有架构设计
- 影响其他项目或系统的决策
- 不可逆或难以回退的变更

#### 3. 资源与权限
- 需要执行 project owner 主机上的脚本（需 `/approve`）
- 需要访问 project owner 的私有数据或服务
- 需要创建/修改 cron job 或其他自动化
- 需要外发消息/邮件/通知

#### 4. 验收与阶段转换
- 阶段完成是否达标
- 能否从 design 进入 implementation
- 能否从 implementation 进入 validation
- 能否正式收尾/归档

#### 5. 安全与风险
- 可能泄露隐私或敏感信息
- 可能破坏现有系统或数据
- 不确定后果的操作

---

### 🟡 条件升级（coordinator agent 判断）

#### 1. 未定义场景
- 协议/文档中没有明确说明的情况
- 先例不足的新问题类型

**判断标准：**
- 如果类似场景在过去 3 个项目中有处理先例 → 🟢 自主处理
- 如果完全无先例，或先例冲突 → 🔴 升级

#### 2. 外部 Agent 调度
- Codex / Claude / Gemini 等进入项目执行具体任务

**判断标准：**
- 如果 scope 和边界已在 COLLAB 中明确 → 🟢 可直接调度
- 如果涉及新项目、新任务类型、或边界不清 → 🔴 先确认

#### 3. 时间敏感的决策
- 需要快速响应，但可能违反常规流程

**处理方式：**
- 先按最保守方案执行（不破坏、不回退）
- 同时标记为 "pending review"
- 在 24h 内提交 project owner review

---

### 🟢 coordinator agent 自主处理

#### 1. 文档维护
- 按已确认结构整理、更新、迁移文档
- 维护 README、STATUS、RESUME、COLLAB
- 更新 docs map 和 navigation

#### 2. 工作态维护
- 更新 RESUME 中的当前状态
- 在 COLLAB 中记录 handoff / writeback
- 标记 work items 的状态变更

#### 3. 信息收集与整理
- 读取项目文件、检索记忆、查询外部信息
- 整理成结构化摘要
- 生成报告或分析

#### 4. 执行已确认的方案
- 按照已拍板的设计进行实现
- 按照已确认的测试方案进行验证
- 按照已定义的格式进行输出

## 升级方式

### 方式一：异步升级（推荐）
适用于非紧急场景：

1. 在 COLLAB.md 的 "Need review / need decision" 区域添加条目
2. 标记优先级和期望响应时间
3. 等待 project owner review

**格式示例：**
```markdown
### REVIEW-2026-04-07-01
- **问题**：是否需要把 skill draft 正式打包成 skill？
- **背景**：当前 skill draft 已有边界、结构、协议参考
- **建议**：暂不打包，继续观察 2-3 个项目
- **期望响应**：24h 内
```

### 方式二：即时升级
适用于需要立即决策的场景：

1. 直接向 project owner 发起对话
2. 简要说明：问题、背景、建议方案、需要拍板的点
3. 等待回复后再继续

### 方式三：紧急标记
适用于时间敏感但不敢擅自决策的场景：

1. 按最保守方案执行（安全优先）
2. 在 COLLAB 中标记为 "EMERGENCY-DECISION-PENDING"
3. 同时尝试联系 project owner
4. 24h 内必须完成 review

## 与 COLLAB.md 的集成

### COLLAB 中的相关字段

```markdown
## Need review / need decision
<!-- 所有待升级事项列在这里 -->

### [ID] 简短描述
- **类型**：方向/边界/资源/验收/异常
- **升级方式**：异步/即时/紧急
- **状态**：waiting / reviewing / decided
- **决策**：(待填写)
```

### 状态流转

```
[发现需要升级]
      ↓
[判断升级级别：🔴🟡🟢]
      ↓
🔴 必须升级 → 选择升级方式 → 写入 COLLAB → 等待决策
🟡 条件升级 → 检查先例 → 有则自主 / 无则升级
🟢 自主处理 → 执行 → 在 COLLAB 记录
      ↓
[决策完成后更新 COLLAB 状态]
```

## 协议演化

本协议会随着实际协作经验更新。更新流程：

1. 发现协议未覆盖的场景
2. 按 "条件升级" 处理（保守优先）
3. 与 project owner review 后确认规则
4. 更新本协议文档
5. 在 COLLAB 中记录协议更新
