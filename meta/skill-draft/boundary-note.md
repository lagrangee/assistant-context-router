# Collab Skill Boundary Note

## 目的
定义 `assistant-context-router` 项目内已形成的“文档驱动 + 多 agent 协作”候选规范，何时才适合升格为跨项目通用 skill；并明确未来 skill 的职责边界，避免过早抽象或抽象失焦。

## 当前结论
当前项目内已经形成可复用雏形，但**尚未满足直接升格为通用 skill 的条件**。

当前更准确的定位是：
- 已验证：项目内文档驱动协作 workflow
- 已形成：候选 request/reply 结构、文档分层、协作面板机制
- 尚未完成：跨项目、跨 agent 类型、跨任务类型的稳定验证

## 为什么现在还不能直接做成正式 skill
1. 当前验证样本仍主要集中在单项目：`assistant-context-router`
2. 当前外部 agent 协作样本仍偏重 Codex，尚未充分覆盖 Claude Code、Gemini 等 agent
3. 当前流程更偏向“策略评审 / 文档收口 / step handoff”，尚未证明对实现型、运维型、研究型任务都同样稳定
4. 当前文档治理规则仍是项目内候选方案，尚未经过跨项目复用压力测试

## 升格为通用 skill 的最低条件
只有在以下条件大体满足后，才建议从“项目内候选规范”升格为“跨项目通用 skill”：

### 1. 多项目验证
至少在两个以上项目中跑通，而不是只在 `assistant-context-router` 内自洽。

### 2. 多 agent 验证
至少验证过两类以上外部 agent 协作，而不是只适配单一 agent 风格。

### 3. 多任务类型验证
至少覆盖：
- 策略/规划型
- 实现/编码型
- 评审/收口型

### 4. 文档职责稳定
已经证明下面这些职责分工不会频繁变形：
- `collab.md` 负责当前协作状态
- `plan/` 负责稳定结论
- `progress/` 负责阶段进展与收尾
- `docs/README.md` 负责索引

### 5. 升格边界清楚
已经能够明确区分：
- 哪些是通用 skill 应提供的协作协议
- 哪些仍应留在具体项目文档中

## 未来通用 skill 应负责什么
如果未来升格为通用 skill，建议 skill 只负责**协作协议与文档驱动方法**，而不负责具体项目内容。

### In scope
- 何时启用文档驱动协作
- 如何建立最小协作面板
- request / reply 的推荐结构
- 文档分层的最小规则
- step strategy / step closure 的最小约定
- 人类 / 主助手 / 外部 agent 的角色分工
- 何时应将过程结论升格为正式文档

### Out of scope
- 具体项目的业务策略内容
- 具体项目的目录结构强约束
- 某个 agent 专属的实现细节
- 某个项目的固定文档模板全集
- 自动 writeback / 自动归档实现
- 任何具体 router / orchestration 产品逻辑

## 未来通用 skill 不应取代什么
这个 skill 不应取代：
- 项目自己的 `README.md`
- 项目自己的 `plan/` 文档
- 项目自己的 `progress/` 文档
- 项目自己的业务/架构决策文档

它应该做的是：
> 给另一个 agent 一个清晰的协作方法与文档治理起手式，而不是替项目写完所有内容。

## 触发条件（候选）
未来如果做成通用 skill，触发语境应类似：
- 用户要和 Codex / Claude Code / Gemini / 其他外部 agent 协作
- 一个任务需要通过文档而不是聊天记录来交接
- 一个项目已经开始出现 step 文档、progress 文档、协作面板、文档索引需求
- 用户明确要求建立可复用的多 agent 协作协议或文档治理机制

## 建议的产出形式
未来 skill 最适合产出的不是“大量说明文字”，而是：
1. 一套简洁的协作协议说明
2. 一套最小文档分层约定
3. 一个最小协作面板模板
4. 一个最小 docs map 模板
5. 一个“何时升级为正式文档 / 何时归档”的判断规则

## 当前项目的下一步
在 `assistant-context-router` 内，当前更合适的动作不是立刻创建 skill，而是：
1. 继续用当前机制推进 Step 2 策略评审与 validation design
2. 继续观察这套协作方式在更多回合中的稳定性
3. 在后续出现第二个项目样本后，再决定是否由 coordinator agent 主导创建通用 skill

## 当前判断
结论很简单：

> 这套机制已经值得按“未来通用 skill 候选”来设计，
> 但当前还不值得正式打包成 skill。
